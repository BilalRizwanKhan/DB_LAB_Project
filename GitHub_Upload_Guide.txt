1. INTRODUCTION - WHAT IS GITHUB?
================================================================================

GitHub is a cloud-based platform for storing and managing code using Git version
control. It allows you to:

• Backup your code in the cloud
• Track changes and version history
• Collaborate with other developers
• Share your portfolio with employers
• Deploy applications from your repository

Your DreamHomes project will be uploaded to GitHub, making it accessible from
anywhere and safe from local computer failures.

3. METHOD 1: USING VISUAL STUDIO CODE (RECOMMENDED)
================================================================================

This is the most beginner-friendly method with a good balance of power and ease.

STEP 1: INSTALL REQUIRED SOFTWARE
----------------------------------
1. Install Git from https://git-scm.com/download/win
2. Install VS Code from https://code.visualstudio.com/
3. Restart your computer after installation

STEP 2: OPEN YOUR PROJECT IN VS CODE
-------------------------------------
1. Launch Visual Studio Code
2. Click "File" → "Open Folder"
3. Navigate to: C:\Users\SCS\Downloads\realestate
4. Click "Select Folder"

STEP 3: INITIALIZE GIT REPOSITORY
----------------------------------
1. Look at the left sidebar, click the "Source Control" icon (looks like a branch)
2. Click "Initialize Repository" button
3. VS Code will create a hidden .git folder in your project

Alternative method:
- Open Terminal in VS Code: View → Terminal (or Ctrl + `)
- Type: git init
- Press Enter

STEP 4: STAGE YOUR FILES
-------------------------
In the Source Control panel, you'll see all your project files listed under
"Changes".

To stage all files:
- Hover over "Changes" section
- Click the "+" icon that appears
- All files move to "Staged Changes"

To stage specific files:
- Click the "+" icon next to individual files

IMPORTANT: Check your .gitignore file
- VS Code should respect your .gitignore automatically
- Files like venv/, .env, uploads/ should NOT appear in the changes list
- If you see these folders, your .gitignore is working correctly

STEP 5: MAKE YOUR FIRST COMMIT
-------------------------------
1. At the top of Source Control panel, find the message box
2. Type a commit message: "Initial commit - DreamHomes real estate platform"
3. Click the "✓ Commit" button (or press Ctrl + Enter)
4. Your changes are now saved locally

STEP 6: CREATE A GITHUB REPOSITORY
-----------------------------------
1. Go to https://github.com/
2. Log in to your account
3. Click the "+" icon (top right) → "New repository"
4. Fill in the details:
   - Repository name: dreamhomes-realestate (or your preferred name)
   - Description: "Full-featured real estate platform with FastAPI backend"
   - Choose: Public (visible to everyone) or Private (only you can see)
   - DO NOT initialize with README (you already have files)
5. Click "Create repository"

STEP 7: CONNECT VS CODE TO GITHUB
----------------------------------
GitHub will show you a page with setup instructions. Copy the repository URL.
It looks like: https://github.com/yourusername/dreamhomes-realestate.git

In VS Code Terminal (View → Terminal):

1. Add the remote repository:
   git remote add origin https://github.com/yourusername/dreamhomes-realestate.git

   Replace "yourusername" with your actual GitHub username

2. Verify the remote was added:
   git remote -v

   You should see the URL listed twice (fetch and push)

STEP 8: PUSH YOUR CODE TO GITHUB
---------------------------------
In VS Code Terminal, type:

git push -u origin main

If you get an error about "main" branch, try:
git branch -M main
git push -u origin main

AUTHENTICATION:
When prompted for credentials, you have two options:

OPTION A: Personal Access Token (Recommended)
1. Go to: https://github.com/settings/tokens
2. Click "Generate new token" → "Generate new token (classic)"
3. Name it: "DreamHomes Upload"
4. Set expiration: 90 days (or your preference)
5. Check scopes: "repo" (full control of private repositories)
6. Click "Generate token"
7. COPY THE TOKEN IMMEDIATELY (you won't see it again)
8. When VS Code asks for password, paste the token (not your GitHub password)

OPTION B: GitHub CLI Authentication
1. Install GitHub CLI: https://cli.github.com/
2. In terminal: gh auth login
3. Follow the prompts to authenticate

STEP 9: VERIFY UPLOAD SUCCESS
------------------------------
1. Go to your GitHub repository page
2. Refresh the browser
3. You should see all your project files
4. Click through folders to verify everything uploaded
5. Check that venv/, .env, and uploads/ are NOT present (gitignore worked)

FUTURE UPDATES:
---------------
After making changes to your project:

1. Save your files in VS Code
2. Go to Source Control panel
3. Stage changes (click "+" on modified files)
4. Write commit message
5. Click "✓ Commit"
6. Click "..." menu → "Push" (or press Ctrl+Shift+P → "Git: Push")

4. METHOD 2: USING GIT COMMAND LINE (TRADITIONAL METHOD)
================================================================================

This method uses Git Bash or Windows Command Prompt. More powerful but less
visual than VS Code.

STEP 1: OPEN GIT BASH
----------------------
1. Press Windows key
2. Type "Git Bash"
3. Press Enter

Alternative: Use Command Prompt or PowerShell

STEP 2: NAVIGATE TO YOUR PROJECT
---------------------------------
cd /c/Users/SCS/Downloads/realestate

Verify you're in the right place:
ls

You should see files like: main.py, models.py, README.md, etc.

STEP 3: INITIALIZE GIT REPOSITORY
----------------------------------
git init

This creates a hidden .git folder in your project directory.

STEP 4: CHECK STATUS
--------------------
git status

This shows all files that will be tracked. You should see your project files
but NOT venv/, .env, or uploads/ (thanks to .gitignore).

STEP 5: STAGE ALL FILES
------------------------
git add .

The "." means "add all files in current directory"

To stage specific files only:
git add main.py models.py README.md

Verify staging:
git status

Files should now be listed in green under "Changes to be committed"

STEP 6: MAKE YOUR FIRST COMMIT
-------------------------------
git commit -m "Initial commit - DreamHomes real estate platform"

The -m flag adds a commit message inline.

STEP 7: CREATE GITHUB REPOSITORY
---------------------------------
1. Open your browser
2. Go to https://github.com/
3. Log in
4. Click "+" (top right) → "New repository"
5. Name: dreamhomes-realestate
6. Description: "Full-featured real estate platform"
7. Choose Public or Private
8. DO NOT check "Initialize with README"
9. Click "Create repository"

STEP 8: CONNECT LOCAL TO GITHUB
--------------------------------
Copy your repository URL from GitHub. It looks like:
https://github.com/yourusername/dreamhomes-realestate.git

In Git Bash:
git remote add origin https://github.com/yourusername/dreamhomes-realestate.git

Verify:
git remote -v

STEP 9: RENAME BRANCH TO MAIN (if needed)
------------------------------------------
GitHub uses "main" as default branch name. Your local might be "master".

Check current branch:
git branch

If it shows "master", rename it:
git branch -M main

STEP 10: PUSH TO GITHUB
------------------------
git push -u origin main

The -u flag sets "origin main" as default, so future pushes just need: git push

AUTHENTICATION:
When prompted for username and password:
- Username: your GitHub username
- Password: Personal Access Token (NOT your GitHub password)

TO CREATE PERSONAL ACCESS TOKEN:
1. GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
2. "Generate new token (classic)"
3. Name: "DreamHomes CLI Access"
4. Expiration: 90 days
5. Scope: Check "repo"
6. Generate and COPY the token
7. Use this as your password

STEP 11: VERIFY SUCCESS
------------------------
1. Refresh your GitHub repository page
2. All your files should now be visible
3. Click "commits" to see your commit history

FUTURE WORKFLOW:
----------------
After making changes:

git status                    # See what changed
git add .                     # Stage all changes
git commit -m "Description"   # Commit with message
git push                      # Upload to GitHub

6. COMMON ISSUES & TROUBLESHOOTING
================================================================================

ISSUE 1: "fatal: not a git repository"
---------------------------------------
PROBLEM: You're not in your project directory or Git isn't initialized
SOLUTION:
- Navigate to correct folder: cd /c/Users/SCS/Downloads/realestate
- Initialize Git: git init

ISSUE 2: Authentication Failed / Password Doesn't Work
-------------------------------------------------------
PROBLEM: GitHub no longer accepts account passwords for Git operations
SOLUTION: Use Personal Access Token instead
1. Go to: https://github.com/settings/tokens
2. Generate new token (classic)
3. Copy the token
4. Use token as password when Git asks for credentials

ISSUE 3: "remote origin already exists"
----------------------------------------
PROBLEM: You tried to add remote twice
SOLUTION:
- View current remote: git remote -v
- Remove it: git remote remove origin
- Add correct one: git remote add origin YOUR_GITHUB_URL

ISSUE 4: "failed to push some refs"
------------------------------------
PROBLEM: Remote repository has changes you don't have locally
SOLUTION:
- Pull first: git pull origin main
- Then push: git push origin main

ISSUE 5: Accidentally Committed .env File
------------------------------------------
PROBLEM: Sensitive data (passwords, API keys) pushed to GitHub
SOLUTION (URGENT):
1. Remove from Git history: git rm --cached .env
2. Add to .gitignore: echo ".env" >> .gitignore
3. Commit: git commit -m "Remove .env from tracking"
4. Push: git push
5. IMPORTANT: Change all passwords/keys in the .env file (they're compromised)

ISSUE 6: "src refspec main does not match any"
-----------------------------------------------
PROBLEM: No commits made yet, or branch named differently
SOLUTION:
- Make sure you committed: git commit -m "Initial commit"
- Check branch name: git branch
- If it's "master": git branch -M main
- Then push: git push -u origin main

ISSUE 7: Large Files Upload Failed
-----------------------------------
PROBLEM: Files over 100MB rejected by GitHub
SOLUTION:
- Add to .gitignore if not needed
- Use Git LFS for large files: https://git-lfs.github.com/
- Or store large files elsewhere (cloud storage, CDN)

ISSUE 8: venv/ or __pycache__/ Showing in Changes
--------------------------------------------------
PROBLEM: .gitignore not working properly
SOLUTION:
- If already tracked: git rm -r --cached venv/ __pycache__/
- Verify .gitignore has these lines:
  venv/
  __pycache__/
  *.pyc
- Commit: git commit -m "Remove cached files"
- Push: git push

ISSUE 9: Merge Conflict
------------------------
PROBLEM: Same file edited in two places
SOLUTION:
1. Open conflicted file
2. Look for conflict markers: <<<<<<< ======= >>>>>>>
3. Manually choose which changes to keep
4. Remove conflict markers
5. Save file
6. Stage: git add filename
7. Commit: git commit -m "Resolve merge conflict"

ISSUE 10: Forgot Commit Message
--------------------------------
PROBLEM: Git opened text editor (Vim) for commit message
SOLUTION IN VIM:
1. Press "i" to enter insert mode
2. Type your message
3. Press "Esc"
4. Type ":wq" and press Enter

PREVENTION:
Always use: git commit -m "Your message here"


================================================================================
7. BEST PRACTICES
================================================================================

COMMIT MESSAGE GUIDELINES:
--------------------------
GOOD:
- "Add user authentication with JWT"
- "Fix property search filter bug"
- "Update README with installation steps"

BAD:
- "update"
- "changes"
- "fixed stuff"
- "asdfghjkl"

Use present tense: "Add feature" not "Added feature"
Be specific about WHAT you changed and WHY

FILES TO NEVER COMMIT (YOUR PROJECT):
-------------------------------------
Your .gitignore already handles these, but verify:

✗ venv/ or env/          - Virtual environment (huge, machine-specific)
✗ .env                   - Contains SECRET_KEY, database passwords
✗ __pycache__/           - Python bytecode (auto-generated)
✗ *.pyc, *.pyo          - Compiled Python files
✗ uploads/               - User-generated content (privacy concern)
✗ *.db, *.sqlite        - Database files (use migrations instead)
✗ .vscode/, .idea/      - IDE settings (personal preferences)
✗ *.log                 - Log files (temporary data)

✓ requirements.txt      - KEEP THIS (lists dependencies)
✓ .env.example          - KEEP THIS (template without secrets)
✓ README.md             - KEEP THIS (project documentation)
✓ All .py files         - KEEP THIS (your code)
✓ frontend/             - KEEP THIS (HTML/CSS/JS)
✓ database_schema.sql   - KEEP THIS (database structure)

COMMIT FREQUENCY:
-----------------
Commit when you:
- Complete a feature
- Fix a bug
- Reach a stable state
- Before trying risky changes

DON'T commit:
- Every single line change
- Broken/untested code
- In the middle of a feature

BRANCH STRATEGY (ADVANCED):
----------------------------
For now, work on "main" branch. Later, learn about:
- Feature branches: git checkout -b feature-name
- Pull requests for code review
- Protected branches

SECURITY REMINDERS:
-------------------
1. NEVER commit .env files
2. NEVER commit passwords or API keys
3. Check your .gitignore before first commit
4. Use environment variables for secrets
5. Rotate credentials if accidentally committed
6. Make repository private if it contains sensitive code

BACKUP STRATEGY:
----------------
GitHub IS your backup, but also:
- Keep local copy on your computer
- Don't delete local repo after pushing
- Consider cloning to external drive monthly

git restore filename                    # Discard changes in file
git restore --staged filename           # Unstage file
git reset --soft HEAD~1                 # Undo last commit (keep changes)
git reset --hard HEAD~1                 # Undo last commit (DELETE changes)
